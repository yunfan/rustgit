# 经验总结：rustgit 存储架构重构

## 重大教训

1. **借用生命周期与返回值重构带来的蝴蝶效应**：
   在把存储引擎由具体类型 `LiteMap`（内存常驻）重构为通过 Trait 返回泛型（特别是涉及磁盘读取时的分配问题）的过程中，原先的 `Option<&[u8]>` 不再适用。为了处理临时生成并装箱的磁盘对象内容，不得不将 `get_as` 等接口改为返回 `Option<Box<[u8]>>`。这带来了一连串蝴蝶效应：诸如 `read_text` 这样的高层封装不再能简单返回 `Result<&str>`，因为无法借用函数内部临时生成的智能指针。对于架构设计，这提示我们在早期如果预期到外部数据的存在（磁盘），需要从一开始避免过度使用切片引用作为业务返回层结构，要么使用 `Cow`，要么果断使用装箱/拥有的结构。

2. **全局状态的非预期破坏（Git 仓库本身的 .git）**：
   在进行磁盘测试，执行 `rustgit-cmd init . --storage disk` 的时候，由于当前工作目录恰好是真实开发所在的 `rustgit` 仓库，该命令由于写盘功能激活，直接重置了包含提交指针等重要参数在内的真实 `.git/config` 文件及 `HEAD` 文件。
   **规避方案**：任何涉及环境状态的 I/O 写操作，尤其是在开发自身具备类似功能的工具时，**切忌**在当前的系统工程目录进行真实测试验证。应严格限制测试运行目录在隔离的 `/tmp` 内或创建 `test_repo_env`。

3. **依赖锁定对于本地依赖测试的阻碍**：
   子工程 `rustgit-cmd` 原来依赖于向上的 GitHub 源。如果在父级工程（如核心库）尚未合并 PR 时需要测试 CLI ，直接执行构建由于网络锁的存在无法链接到刚开发的本地代码。必须在开发调试期使用 `path` 在 `Cargo.toml` 做指向重定向，或者采用 Workspace 配置统筹管理，以便缩短开发与联合调试的链路并尽早暴露类型错误。
